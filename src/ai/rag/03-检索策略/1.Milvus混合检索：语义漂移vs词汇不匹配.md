 # 混合检索：语义漂移 vs 词汇不匹配

在做 RAG 或知识库检索时，很多人都会踩一个坑：

> 向量检索很聪明，但经常答非所问
> 关键词检索很精准，但经常啥也找不到

这篇笔记用**真实场景**彻底讲清楚两个核心问题：

- 语义漂移（Semantic Drift）
- 词汇不匹配（Vocabulary Mismatch）

并解释为什么 **Hybrid 检索是工业界标准方案**。

---

# 一句话先记住

| 问题    | 谁会犯错 |
| ----- | ---- |
| 语义漂移  | 向量检索 |
| 词汇不匹配 | BM25 |

---

# 一、语义漂移：向量检索的大坑

## 本质

向量检索擅长找 **同一话题**
但用户需要的是 **同一个问题**

> 话题相关 ≠ 问题相关

这就是语义漂移。

---

## 场景1：电商客服真实翻车案例

用户问：

```
我的订单什么时候发货？
```

知识库有这些文档：

1. 发货时效说明：普通商品 24–48 小时内发货
2. 订单取消规则：未发货订单可以申请取消
3. 修改收货地址：发货前可以修改地址
4. 仓库发货流程：每天 18:00 出库
5. 如何查询订单：进入 App 查看订单

### 向量检索可能给出的相似度

| 文档     | 相似度  |
| ------ | ---- |
| 订单取消规则 | 0.86 |
| 修改地址   | 0.84 |
| 发货流程   | 0.82 |
| 发货时效   | 0.80 |

真正答案 **排第4**。

用户问"什么时候发货"，系统却先告诉他：

> 未发货订单可以取消

这就是典型 **语义漂移**。

向量找到了"订单流程话题"，但不是用户真正的问题。

---

## 场景2：程序员文档

用户问：

```
Java 内存溢出怎么解决？
```

知识库：

| 文档     | 内容       |
| ------ | -------- |
| GC调优指南 | JVM 垃圾回收 |
| 线程优化   | 线程池实践    |
| OOM排查  | 内存溢出分析   |
| 性能优化   | 系统调优     |

向量可能排序：

1. GC调优
2. 线程优化
3. 性能优化
4. OOM排查（真正答案）

原因：

向量学的是 **主题相关性（topical similarity）**
用户需要的是 **问题相关性（problem relevance）**

# 二、词汇不匹配：BM25 的大坑

BM25 的原则：

> 只认字面，不认意思

---

## 场景1：同义词灾难

用户问：

```
怎么查物流？
```

文档写：

```
查看配送进度的方法
```

BM25 分词结果：

| Query | 文档   |
| ----- | ---- |
| 查     | 查看 ❌ |
| 物流    | 配送 ❌ |
| 物流    | 进度 ❌ |

结果：**0 命中**

用户：？？？

---

## 场景2：用户语言 vs 官方语言

用户：

```
怎么退款？
```

文档：

```
申请售后流程说明
```

BM25：

```
退款 ≠ 售后
```

→ 搜不到。

---

## 场景3：口语 vs 书面语

用户：

```
东西坏了咋整
```

文档：

```
商品质量问题处理流程
```

BM25 直接失效。

---

## 场景4：中英文混用

用户：

```
API 限流怎么办
```

文档：

```
接口调用频率限制说明
```

BM25：

```
API ≠ 接口
限流 ≠ 频率限制
```

再次失败。

---

# 三、把两个灾难放一起看

用户问：

```
订单还没到怎么办
```

知识库：

| 文档     | BM25 | 向量 |
| ------ | ---- | -- |
| 物流延迟处理 | ❌    | ✅  |
| 退货政策   | ❌    | ✅  |
| 发货时效   | ❌    | ✅  |
| 订单取消   | ❌    | ✅  |

结果：

- BM25 → 找不到
- 向量 → 找到一堆"相关话题"，但排序可能错误

单独用任何一个都会失败。

---

# 四、Hybrid 检索如何救场

Hybrid = 向量搜索 + BM25 搜索 + RRF 融合

当 query 包含：

```
订单号 2026012345
```

BM25：

> 精确命中唯一结果

向量：

> 找到所有相关流程

融合后：

1. 精确答案排第一
2. 语义相关排后面

完美互补。

---

# 五、RRF 排序为什么有效

RRF 公式：

```
score = 1 / (k + rank)
```

只看 **排名**，不看分数。

原因：

| 搜索方式 | 分数范围  |
| ---- | ----- |
| 向量   | 0~1   |
| BM25 | 0~10+ |

无法直接加权。

RRF 的聪明点：

> 不比较分数，只比较排名

因此非常鲁棒。

# 六、一个终身记住的比喻

找医生：

| 方法     | 类比        |
| ------ | --------- |
| BM25   | 按名字找医生    |
| 向量     | 按科室找医生    |
| Hybrid | 先找名字，再看科室 |

只用一个都会出事。

# 七、工业界结论

当知识库规模 > 1 万条：

| 方案     | 结果     |
| ------ | ------ |
| 纯 BM25 | 召回灾难   |
| 纯向量    | 排序灾难   |
| Hybrid | 唯一可用方案 |

这不是高级技巧，而是：

> **RAG 的入场券**
