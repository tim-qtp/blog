## 为什么要单独设计文档摄取层

在 RAG 系统里，很多问题并不是出在“模型不会回答”，而是出在“文档没有被正确理解”。

原始文档通常有这些问题：

- 格式很多：PDF、Word、Excel、HTML、图片、邮件、音视频
- 结构复杂：标题、正文、表格、注释、页眉页脚、目录
- 内容混杂：文字、图片、表格、公式、脚注会同时出现
- 排版不稳定：多栏、旋转页、扫描件、截图、低清图片

所以，摄取层的职责不是简单把文件转成文本，而是尽量还原文档语义结构，再把它变成适合检索的中间表示。

你可以把它理解成两步：

1. **文档理解**：识别内容是什么、在哪里、和谁相邻
2. **知识整理**：把理解后的内容切成可索引、可召回的知识块

---

## 一条完整的数据流是什么样的

从系统视角看，RAGFlow 的文档摄取流程大致如下：

```text
上传文件
  -> 识别文件类型
  -> 选择解析器
  -> 提取文本 / 图片 / 表格 / 版面信息
  -> 生成统一中间结构
  -> 进行清洗、去目录、补充元信息
  -> chunk 切分
  -> 入库、建索引
  -> 检索和问答
```

这个链路中最重要的一步是“生成统一中间结构”。因为只有先把异构文档统一起来，下游的 chunker、索引器、检索器才能用同一套逻辑工作。

## 核心模块分别负责什么

从代码组织上看，几个关键模块大致承担以下职责：

- `rag/flow/parser/`：统一解析入口，负责按文件后缀分发
- `deepdoc/`：底层文档视觉理解能力，例如 OCR、版面分析、表格结构识别
- `rag/flow/chunker/`：把结构化结果切成更适合检索的 chunk
- `rag/flow/pipeline.py`：把解析、切分、入库流程串起来
- `api/db/services/`：管理文件、文档、任务、知识库状态

对应源码入口可直接跳转到这里：

- `Parser` 入口：[`rag/flow/parser/parser.py`](./rag/flow/parser/parser.py)
- PDF 结构解析：[`deepdoc/parser/pdf_parser.py`](./deepdoc/parser/pdf_parser.py)
- Word 解析：[`rag/app/naive.py`](./rag/app/naive.py)
- Chunk 切分：[`rag/flow/chunker/token_chunker.py`](./rag/flow/chunker/token_chunker.py)

一个简化的关系图可以这样理解：

```text
文件类型判断
  -> Parser
  -> DeepDoc / OCR / VLM / 专用解析器
  -> 标准化 JSON / Markdown / Text
  -> Chunker
  -> Index / Retrieval
```

## 不同文件类型的解析策略对比

RAGFlow 的摄取层并不是“一个解析器走天下”，而是会针对不同文件类型选择不同的解析策略。这个设计的本质是：**先理解文件的天然结构，再决定怎么把它转成知识块**。

下面把最典型的四类内容单独展开讲，并且把它们最终如何进入统一摄取链路说明清楚。

### 1. PDF：最复杂，也最值得重点优化

PDF 的问题不在于“能不能读出字”，而在于“读出的字是不是按人类阅读顺序组织起来”。它常见的复杂性包括：

源码入口可直接看 [`Parser._pdf`](./rag/flow/parser/parser.py)，底层版面解析在 [`RAGFlowPdfParser`](./deepdoc/parser/pdf_parser.py)。

- 文本层和图片层并存
- 多栏排版导致阅读顺序错乱
- 扫描件没有可复制文本
- 图表、公式、脚注、页眉页脚混在一起
- 目录页、附录页、空白页可能污染正文

因此 PDF 解析通常是一个组合流程：

1. 先判断是否有可直接抽取的文本层
2. 再判断是否需要 OCR
3. 结合页面布局做阅读顺序重排
4. 对表格、图片、公式单独处理
5. 去掉目录和噪声页，再进入 chunk 切分

#### PDF 示例 1：文本型 PDF

假设一份技术文档中有如下内容：

```text
第 1 章 系统概览
系统由采集层、解析层和索引层组成。
解析层负责提取结构化内容。
```

如果 PDF 本身存在文本层，解析器可以直接拿到这段内容，但仍然要做两件事：

- 保留标题层级
- 让后续 chunk 了解“这段话属于哪个章节”

最终更理想的结构是：

```json
{
  "layout_type": "section",
  "text": "系统由采集层、解析层和索引层组成。解析层负责提取结构化内容。",
  "section": "第 1 章 系统概览",
  "page_number": 1
}
```

#### PDF 示例 2：扫描版 PDF

如果同样一页是扫描件，页面其实是图片：

- OCR 会先识别文字
- 版面分析会识别标题、正文、表格区域
- 如果是双栏，还需要重建阅读顺序

例如一页中左栏是“背景”，右栏是“方案”，直接 OCR 可能得到乱序结果：

```text
方案：我们采用三阶段流水线。
背景：传统方法难以处理复杂版面。
```

正确做法应该是根据坐标重排为：

```text
背景：传统方法难以处理复杂版面。
方案：我们采用三阶段流水线。
```

#### PDF 示例 3：表格型 PDF

表格是 PDF 里最容易被扁平化破坏的对象之一。比如：

| 模块 | 延迟 | 准确率 |
|------|------|--------|
| A    | 120ms | 91% |
| B    | 80ms  | 94% |

如果没有识别表头和单元格关系，输出就可能变成：

```text
A 120ms 91%
B 80ms 94%
```

这样“模块 A 的准确率是多少”这类问题就很难准确回答。

### 2. Word / DOCX：结构稳定，适合做语义分段

Word 文档比 PDF 更接近“作者真正写出来的结构”，因为它通常保留了：

源码入口可看 [`Parser._word`](./rag/flow/parser/parser.py)，其底层文档抽取与 Markdown 转换逻辑主要在 [`rag/app/naive.py`](./rag/app/naive.py)。

- 标题样式
- 段落样式
- 列表层级
- 表格结构
- 图片和图注

因此 Word 的重点不是 OCR，而是**样式语义提取**。

#### Word 示例 1：章节标题与正文

假设文档里有如下层级：

- `1. 总体设计`
- `1.1 架构说明`
- `1.2 模块职责`

解析器如果能识别这些层级，就可以把正文挂到对应章节下：

```json
[
  { "layout_type": "title", "text": "1. 总体设计" },
  { "layout_type": "subtitle", "text": "1.1 架构说明" },
  { "layout_type": "text", "text": "系统采用三层结构：接入层、处理层、服务层。" }
]
```

这会直接提升 chunk 的可读性和召回准确率。

#### Word 示例 2：列表和说明

Word 里经常有这种内容：

- 支持增量更新
- 支持多租户
- 支持知识库权限控制

如果解析时把它们当成普通长文本，后面就很难判断每一条是否独立成立。更好的方式是保留列表关系，甚至把每个列表项当作一个子块。

#### Word 解析的关键点

Word 解析最需要关注的是：

- 样式是否被正确映射为标题/正文/列表
- 表格是否保持单元格结构
- 图片是否和图注绑定
- 是否保留文档顺序和章节顺序

### 3. Excel / CSV：核心不是文本，而是表格语义

Excel 最容易踩坑的地方是：很多系统把它当“纯数据文本”处理，但实际上 Excel 的语义主要来自表格关系。

源码入口可看 [`Parser._spreadsheet`](./rag/flow/parser/parser.py)，默认表格解析走 [`deepdoc/parser/excel_parser.py`](./deepdoc/parser/excel_parser.py) 对应的 `ExcelParser`。

#### Excel 示例 1：简单指标表

| 模块 | 负责人 | 状态 |
|------|--------|------|
| Parser | 张三 | 完成 |
| Chunker | 李四 | 进行中 |

如果只按单元格顺序读取，可能得到：

```text
模块 负责人 状态 Parser 张三 完成 Chunker 李四 进行中
```

但真正有用的知识应该是：

- `Parser` 由 `张三` 负责
- `Chunker` 由 `李四` 负责
- `Chunker` 当前状态是 `进行中`

这就要求解析器保留“行是实体、列是属性”的映射关系。

#### Excel 示例 2：合并单元格

很多报表里会出现合并单元格，例如某一列的大标题覆盖多行。此时如果不展开合并关系，下游会丢失字段归属。

例如：

- `2025 年上半年`
  - `1 月：100`
  - `2 月：120`

如果合并单元格处理不当，可能不知道 `1 月`、`2 月` 属于哪个时间段。

#### Excel/CSV 解析的关键点

- 保留表头和列顺序
- 保留行列对应关系
- 识别合并单元格
- 区分“表格数据”和“说明文字”

对 CSV 来说，虽然没有复杂样式，但仍然要注意：

- 第一行是否是表头
- 分隔符是否异常
- 某些列是否是数值/日期类型

### 4. 图片：OCR 解决文字，VLM 解决语义

图片类输入的价值往往不止是“图片里写了什么”，还包括“图片表达了什么”。因此通常要分成两层：

源码入口可看 [`Parser._image`](./rag/flow/parser/parser.py)，其中 OCR 路径直接走 `deepdoc.vision.OCR`，VLM 路径走多模态模型配置。

- **OCR**：提取可见文本
- **VLM**：描述图片语义、图表关系、页面结构

#### 图片示例 1：截图

假设是一张系统截图，里面显示：

> 当前知识库已完成 128 个文档摄取任务。

OCR 能把这句话提出来，而 VLM 可以进一步识别这是不是控制台截图、报表截图还是告警截图。

#### 图片示例 2：图表

假设图片是一张折线图，图例分别是 `Model A` 和 `Model B`，趋势显示 `Model B` 在后期更稳定。仅 OCR 只能读出坐标轴和图例，VLM 才能补充“谁更稳定、趋势如何变化”这类语义。

#### 图片示例 3：架构图

架构图往往没有太多纯文本，但有大量关系信息，比如：

- 数据从哪里流入
- 哪个模块在前，哪个模块在后
- 哪些服务之间互相调用

这类内容非常适合 VLM 提炼成描述文本，再进入后续 chunk 与检索流程。

#### 图片解析的关键点

- OCR 和 VLM 不要互相替代，而要互补
- 图片块应保留原图引用，方便回看
- 图片内容最好带上页码和上下文
- 图表和截图应区分处理策略

---

### 5. 小结：四类文件的共性目标

虽然 PDF、Word、Excel、图片的解析方法不一样，但最终目标完全一致：

1. 尽可能还原结构
2. 尽可能保留上下文
3. 尽可能把信息变成可检索知识块

换句话说，**解析方式可以不同，但中间结构必须统一**。这样后续的 chunk 切分、向量化、索引和问答才能走同一套逻辑。

## PDF 的技术难点与示例

PDF 之所以最值得单独讲，是因为它经常被误认为“就是文本文件”。实际上，PDF 只是页面描述格式，不保证阅读顺序等于内部存储顺序。

源码位置：[`rag/flow/parser/parser.py`](./rag/flow/parser/parser.py) 中的 `Parser._pdf`，对应底层实现主要在 [`deepdoc/parser/pdf_parser.py`](./deepdoc/parser/pdf_parser.py)。

### 难点 1：阅读顺序不稳定

例如一页双栏论文，PDF 内部可能按“左栏第一段 -> 右栏第一段 -> 左栏第二段”以外的顺序存储。若直接抽文本，结果会乱。

### 难点 2：扫描件没有文本层

扫描版 PDF 实际上是多页图片。此时必须先做 OCR。

### 难点 3：表格和图片不能只靠文本

下面这个例子说明为什么要保留结构。

#### 示例：简单表格

| 年份 | 营收 | 增长率 |
|------|------|--------|
| 2024 | 1200 | 18% |
| 2025 | 1500 | 25% |

如果把它直接变成纯文本，可能只剩下：

```text
2024 1200 18%
2025 1500 25%
```

这样虽然“看得到字”，但失去了：

- 列名
- 行列语义
- 指标对应关系

更好的做法是把它保留成结构化块，或者转成带表头的 Markdown 形式。

### 难点 4：目录页会污染检索

很多 PDF 会在开头有目录页。如果不去掉，检索时可能老是召回目录而不是正文。

### 难点 5：页面位置和引用很重要

在问答场景下，用户经常会问“这个结论在哪一页”。所以解析结果最好保留页码、坐标、段落位置。

## Word、Excel、图片的处理示例

### Word 示例

假设有一个 Word 文档：

- 标题：`第二章 方案设计`
- 小节：`2.1 系统架构`
- 小节：`2.2 数据流`
- 表格：模块职责说明

如果解析器能识别标题样式，那么输出结构可能是：

```json
[
  { "layout_type": "title", "text": "第二章 方案设计" },
  { "layout_type": "section", "text": "2.1 系统架构" },
  { "layout_type": "text", "text": "系统由三层组成：采集层、处理层、检索层。" }
]
```

这比直接输出整篇长文本更适合后续 chunk 切分。

### Excel 示例

假设一个 Excel 表格如下：

| 模块 | 负责人 | 状态 |
|------|--------|------|
| Parser | 张三 | 完成 |
| Chunker | 李四 | 进行中 |

如果转成 Markdown，至少还能保留表头关系：

```markdown
| 模块 | 负责人 | 状态 |
|------|--------|------|
| Parser | 张三 | 完成 |
| Chunker | 李四 | 进行中 |
```

这样检索“谁负责 Chunker”时，系统能更容易召回正确答案。

### 图片示例

假设图片里写着：

> 当前系统支持 OCR、布局分析和表格识别。

OCR 能把这句话提取出来；VLM 还能告诉你这张图是不是流程图、截图还是架构图。最终在结构上，这张图可以被包装为一个独立的媒体块。

```json
{
  "doc_type_kwd": "image",
  "text": "当前系统支持 OCR、布局分析和表格识别。",
  "image": "<image_ref>",
  "page_number": 3
}
```

## 统一中间结构为什么重要

RAG 系统最怕的不是文档多，而是格式多却没有统一表示。

如果每种格式都单独写一套下游逻辑，会出现这些问题：

- 维护复杂
- 检索规则不一致
- 表格和图片处理被忽略
- PDF 和 Word 召回效果不统一

所以摄取层会尽量把所有输入转成同一种“知识对象”。常见字段包括：

- `text`：文本内容
- `image`：图片引用
- `doc_type_kwd`：文本、图片、表格等类型
- `layout_type`：标题、正文、图、表等版面类型
- `page_number`：页码
- `positions`：坐标信息
- `context_above` / `context_below`：上下文

### 示例：统一结构

```json
[
  {
    "layout_type": "title",
    "text": "1. 系统架构",
    "page_number": 1
  },
  {
    "layout_type": "text",
    "text": "系统由解析层、切分层和检索层组成。",
    "page_number": 1
  },
  {
    "layout_type": "table",
    "text": "模块职责表",
    "page_number": 2
  }
]
```

统一结构的意义是：不管原始文件是什么，下游都能把它当作“可检索知识块”处理。

## chunk 切分如何影响检索质量

很多人以为 RAG 效果差是 embedding 模型不够强，实际上经常是 chunk 切分没做好。

### 过大 chunk 的问题

- 噪声多
- 召回后上下文太散
- 模型注意力被稀释

### 过小 chunk 的问题

- 上下文不足
- 段落被切碎
- 答案缺少必要背景

### 更好的切分方式

摄取层通常会结合：

- 标题层级
- 段落边界
- token 长度
- 表格/图片上下文
- 页码和位置

#### 示例：不合理切分

```text
Chunk 1: 系统由解析层组成。
Chunk 2: 切分层和检索层组成。
```

这显然把一句话切坏了。

#### 示例：合理切分

```text
Chunk 1: 系统由解析层、切分层和检索层组成。
```

### 表格和图片为什么需要上下文

如果单独拿出一个表格：

| 指标 | 数值 |
|------|------|
| Accuracy | 92.3% |

用户不知道这是哪个模型、哪个版本、哪个实验条件下的结果。

所以更好的 chunk 应该是：

```text
模型 A 的实验结果如下：
| 指标 | 数值 |
|------|------|
| Accuracy | 92.3% |
```

这就是上下文窗口的价值。

## 配置项如何改变解析行为

摄取层不是一套固定流程，而是可配置系统。不同配置会直接影响解析质量。

在源码里，相关参数主要定义在 [`ParserParam`](./rag/flow/parser/parser.py) 中，尤其是 `setups` 里的 `pdf`、`word`、`spreadsheet`、`image` 这些分支。

### 常见配置影响

#### 1. `parse_method`

决定走哪条解析链路，例如：

- PDF 走纯文本抽取
- PDF 走 OCR
- PDF 走深度文档理解

#### 2. `remove_toc`

是否去除目录页。对检索噪声影响很大。

#### 3. `output_format`

决定输出格式是 JSON、Markdown 还是纯文本。

#### 4. `enable_multi_column`

是否启用多栏重排。对论文和报告类文档尤其重要。

#### 5. `vlm`

是否启用视觉语言模型增强。对图片、图表、扫描件很有用。

### 示例：同一份 PDF，不同配置的效果差异

- 配置 A：`plain_text`
  - 优点：速度快
  - 缺点：结构丢失

- 配置 B：`deepdoc + remove_toc + multi_column`
  - 优点：结构更好，适合知识库
  - 缺点：处理成本更高

## 常见问题与排查示例

### 问题 1：为什么 PDF 顺序乱了？

**可能原因**：

- 多栏文档没有正确识别
- OCR 输出顺序和阅读顺序不一致
- 页面坐标没有保留下来

**排查建议**：

- 检查是否启用了多栏解析
- 看是否保留 `positions`
- 对比原文和解析结果的段落顺序

### 问题 2：为什么检索总是召回目录？

**可能原因**：

- 没有开启目录清理
- 目录页被当正文一起入库

**排查建议**：

- 开启 `remove_toc`
- 检查目录页是否在 chunk 中重复出现

### 问题 3：为什么图片没被理解？

**可能原因**：

- 没有使用 OCR 或 VLM
- 图片块没有进入增强流程

**排查建议**：

- 查看图片解析配置
- 检查是否生成了 `doc_type_kwd=image` 的块

### 问题 4：为什么表格问答不准？

**可能原因**：

- 表格被扁平化
- 表头丢失
- 上下文不足

**排查建议**：

- 确认表格是否保留为结构化块
- 为表格增加上下文窗口
- 优先输出 Markdown 或 JSON

## 总结

从技术角度看，RAGFlow 的文档摄取层不是“读文件”，而是“把文件变成结构化知识”。

它的核心价值在于：

- 统一处理多种文件格式
- 保留文档结构和语义关系
- 通过 OCR、布局识别、表格识别、VLM 协同理解复杂内容
- 将结果转换为适合 chunk 和检索的中间结构

如果说检索层决定“能不能找到答案”，那么摄取层决定“找到的东西是不是对的”。

对于一个 RAG 系统来说，这一层往往是最容易被低估，但也是最能决定最终效果的部分。
